# 插件平台

Szybko 的插件平台让独立能力参与搜索、指令执行、UI、存储和原生工作流，同时由平台控制生命周期、安全边界和用户体验。

## 插件生命周期

平台应支持这些阶段：

- 发现：找到已安装插件并读取声明。
- 注册：记录安装身份、来源、启用状态和信任元数据。
- 编目：向系统其他部分暴露可用插件及其声明能力。
- 索引：将指令和功能转化为可搜索、可执行入口。
- 激活：当用户打开插件功能时创建或复用运行时。
- 展示：将运行时挂载到宿主表面。
- 挂起或隐藏：移除可见展示，同时在有价值时保留运行时。
- 销毁：结束运行时并释放资源。
- 更新或移除：改变插件文件和注册信息，同时保护用户数据。

这些阶段彼此相关，但不应被合并成一个管理器。

## 兼容方向

Szybko 应借鉴 uTools 风格插件中的有用概念：

- Manifest 声明功能。
- 指令和上下文匹配。
- 生命周期钩子。
- 插件 UI 入口。
- 受控的宿主能力访问。
- 本地插件数据。

兼容应分阶段推进。早期兼容可覆盖生命周期、基础指令、简单 UI 和常见 Shell 或剪贴板能力。更高兼容度再补充动态功能、高级匹配类型、更完整存储和更多系统能力。

完全兼容不是首要目标。首要目标是一个干净、快速、安全可演进的本地插件平台。

## 指令平台

插件指令应拥有独立平台区域。指令可能来自 Manifest、运行时注册、用户别名、市场元数据或使用历史。

指令平台应支持：

- 稳定的指令身份。
- 插件和功能关联。
- 用户别名和禁用状态。
- 排序偏好和使用信号。
- 上下文匹配类型。
- 可重建搜索索引。

指令持久化应独立于运行时状态。指令索引可以重建；用户指令偏好应持久化。

## 运行时连续性

有些插件是一次性工具，有些插件是会话。平台应同时支持两者。

对于会话型插件，在 Launcher 和浮动表面之间移动时应保留状态。销毁运行时应是明确行为。隐藏运行时不应意味着卸载、禁用或移除指令。

## 数据与存储

插件数据应按插件身份隔离。平台应区分：

- 插件自有持久数据。
- 插件缓存。
- 平台指令偏好。
- 运行时会话元数据。
- 市场与更新状态。

存储能力应提供足够兼容性支持有用插件，同时为更强隔离和迁移规则留下空间。

## 权限与信任

平台应逐步形成明确的信任等级：

- 内置或第一方插件。
- 本地开发插件。
- 用户安装的可信插件。
- 需要沙箱和审核的市场插件。

原生能力应通过具备权限意识的适配器提供。文件修改、剪贴板自动化、进程控制、屏幕捕获、键盘鼠标自动化等高风险能力，在开放给广泛第三方前需要更强控制。

## 插件市场方向

未来插件市场应负责发现、安装、更新、签名、兼容元数据、权限审查和用户评价。它应建立在插件注册和指令目录之上，而不是替代它们。

本地插件 MVP 不应依赖插件市场。

# 领域模型

本文档定义 Szybko 中长期稳定的产品概念。它描述语义和关系，不描述具体代码形状。

## Plugin

Plugin 是已安装的扩展能力。它可以贡献指令、搜索行为、UI、存储和原生能力使用。插件拥有身份、元数据、声明功能、兼容预期和用户可控制的启用状态。

插件身份应跨版本稳定。运行时身份和指令身份可以从插件身份派生，但不能替代插件身份。

## Manifest

Manifest 是插件对自身能力的声明。它描述入口、展示元数据、功能、指令、运行时偏好和兼容预期。

Manifest 是目录和索引的来源材料，但不是完整事实。用户偏好、动态功能、市场元数据和安装状态应存放在其他领域对象中。

## Feature

Feature 是插件暴露给用户的功能点。它可以打开插件 UI、处理指令、响应搜索上下文、处理选中文件，或执行由原生能力支撑的动作。

Feature 连接插件作者的能力声明和用户可发现的指令。

## Command

Command 是可调用入口，来源可以是插件功能、内置动作、用户别名、动态注册或用户自定义。

Command 会被索引用于发现。它可以是精确指令、模糊别名、上下文匹配、正则匹配、文件匹配、图像匹配，或未来由 AI 路由的工具入口。

Command 的持久化独立于运行时持久化。一个指令可以在任何插件运行时创建之前就存在。

## Runtime

Runtime 是活跃的插件会话。它拥有执行状态、消息通信、生命周期状态和插件侧内存。

一个插件可以有零个、一个或多个 Runtime。当用户预期会话连续时，Runtime 状态应在宿主迁移中保留。

## Runtime Host

Runtime Host 是可以展示 Runtime 的表面。Launcher、浮动窗口、侧边栏、分屏和停靠区域都是宿主例子。

Host 负责展示相关问题：可见性、焦点、布局、窗口行为和展示面控制。它不负责插件能力或指令注册。

## Shell

Shell 是意图输入和动作选择的主界面。它展示搜索结果、处理键盘导航、展示活跃插件表面，并将执行委托给平台服务。

Shell 应保持可替换。它不是平台本身。

## Search Result

Search Result 是用户可查看或执行内容的排序展示。它可以来自内置提供者、指令索引、插件运行时或原生适配器。

Search Result 应携带足够的执行信息，但不应成为文件、应用、指令、插件或适配器响应的通用领域模型。

## Action

Action 是用户选择或工作流产生的可执行意图。它可以打开文件、启动应用、激活插件功能、写入剪贴板、调用原生适配器，或路由到另一个指令。

Action 应足够明确，以便审计、权限检查和路由。

## Adapter

Adapter 是通向操作系统能力的中介桥梁。它暴露领域概念，而不是把平台实现细节泄漏到应用其他部分。

Adapter 让原生行为更容易测试、替换和权限控制。

## User Data

User Data 包括应用偏好、插件数据、指令自定义、使用历史、允许保存的凭据和市场状态。

插件自有数据应按插件身份隔离。平台共享数据应有清晰 schema 和迁移规则。

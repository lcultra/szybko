# 架构

Szybko 围绕一组稳定的平台边界组织。每个边界只拥有一个领域职责，并通过明确契约与其他边界协作。

## 系统边界

### Shell

Shell 是用户交互面。它负责输入、结果展示、选择、轻量窗口控制，以及搜索模式和插件模式之间的切换。

Shell 不负责插件生命周期、原生系统访问或插件安装。它向平台请求搜索、执行、打开运行时、隐藏运行时，或移动运行时到其他展示面。

### 插件平台

插件平台负责已安装插件的身份和声明能力。它回答哪些插件可用、哪些被启用、暴露了哪些指令、应该如何启动等问题。

平台应区分持久化的插件注册信息和当前可用插件目录。安装状态、用户偏好、动态指令和运行时状态不应被合并成一个大对象。

### 运行时系统

运行时系统负责活跃的插件会话。运行时代表一个正在运行的插件实例，拥有执行上下文、生命周期状态、缓存和消息通道。

运行时身份不同于插件身份。一个插件可能没有运行时、只有一个运行时，或拥有多个运行时，这取决于插件设置和用户行为。

### 宿主系统

宿主系统负责运行时显示在哪里。运行时宿主可以是 Launcher、浮动窗口，或未来的侧边栏、分屏、停靠面板。

运行时在宿主之间移动时应保留状态。宿主变化是展示变化，不是插件重启。

### 指令与搜索系统

指令与搜索系统将用户意图转化为可排序结果和可执行动作。它组合内置搜索、插件声明指令、动态指令、最近使用、别名和上下文匹配。

搜索结果是展示与执行契约，不应成为所有原生适配器数据的内部模型。

### 原生适配器层

原生适配器层通过领域概念暴露操作系统能力，例如文件、应用、剪贴板项、显示器、窗口和 Shell 动作。

适配器隔离平台差异。插件和 Shell 应消费稳定能力，而不是直接依赖操作系统实现细节。

### 持久化

持久化包含多类状态：

- 应用配置。
- 插件安装、来源和启用状态。
- 插件自有数据。
- 指令偏好、别名、禁用状态和排序提示。
- 必要时的运行时恢复或会话元数据。
- 插件市场和更新元数据。

这些数据的生命周期不同，不应共享一个无边界的数据容器。

## 数据流

常规搜索流程：

1. Shell 接收用户输入。
2. 搜索系统查询内置提供者、指令索引和符合条件的插件运行时。
3. 结果流式返回 Shell。
4. 用户选择一个动作。
5. 平台将动作路由给对应的执行器、运行时、插件或原生适配器。

常规插件 UI 流程：

1. Shell 或指令请求打开插件功能。
2. 运行时系统创建或复用运行时。
3. 宿主系统将运行时挂载到合适展示面。
4. 插件收到激活上下文。
5. 后续隐藏、分离、移动、挂起或销毁操作更新运行时和宿主状态，但不混淆插件身份。

## 架构规则

- 插件发现不是运行时生命周期。
- 运行时生命周期不是窗口管理。
- 窗口管理不是插件业务逻辑。
- 宿主迁移不是插件重装，也不是运行时重建。
- 搜索索引不是插件注册。
- 原生适配器输出的是领域数据，不是 UI 搜索数据。
- 蓝图级决策必须独立于当前包结构。
